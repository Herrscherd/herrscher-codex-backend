# Codex backend review — 2026-07-15

Scope: the whole standalone Go module, including CI, public API, protocol
adapter, tests, security-sensitive process handling, and documentation.

## Confirmed and fixed

- Added GitHub Actions CI for `gofmt`, `go vet`, race-enabled tests, and build.
- Fixed parsing of the current app-server item types `agentMessage` and
  `commandExecution`; the previous snake_case-only parser could return an empty
  oneshot response and omit tool events.
- Fixed JSON-RPC turn errors being ignored until EOF.
- Added regression tests for both protocol cases and documented the curated
  model-preset limitation.
- `govulncheck ./...` reports no known vulnerabilities.

## Findings retained as proposals

### Medium — request context is not wired into a running turn

`Respond(ctx, ...)` receives a per-request context, but the stream process and
blocking JSONL read use the context captured by `NewBackend`. Cancellation or a
deadline on one request therefore does not interrupt an in-flight turn. Fixing
this requires an explicit read-loop cancellation design so it does not race the
next serialized turn; it is intentionally not changed in this safe-fix pass.

### Medium — model presets are static

`CommandPresets` is synchronous and currently exposes a curated list. The local
Codex 0.144.4 model catalog contains newer/different entries. The README now
points operators to `codex debug models`; a future API could make presets
runtime-discovered or versioned rather than silently assuming a catalog.

### Low — CLI protocol compatibility is not pinned

The app-server schema is generated by the installed Codex CLI and may evolve.
The adapter ignores unknown notifications, but it does not negotiate or pin a
protocol version. A compatibility matrix or a supported CLI-version policy
would make upgrades safer.

### Low — `Verbose` is compatibility-only

The field is retained to match the sibling backend's configuration surface.
The current implementation does not change output behavior based on it; the
README states that app-server diagnostics already go to stderr.

## Areas with no confirmed defect

- Command execution uses `exec.CommandContext`, not a shell, so prompt text is
  not shell-interpreted.
- Memory fence variants are defanged before recalled context is inserted.
- No dependency vulnerability was reported by `govulncheck`.
- Race-enabled tests pass; no data race was reproduced.
- The repository had no existing CI, so there was no failing historical check
  to preserve or diagnose.
